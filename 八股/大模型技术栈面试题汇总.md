## 一、Agent设计与架构

### 1.1 基础概念

#### Q1: 介绍auto deployer的实现难点

**解答：**

Auto Deployer（自动部署系统）的实现难点主要包括：

| 难点 | 说明 | 解决方案 |
|------|------|---------|
| **环境一致性** | 开发/测试/生产环境差异 | 容器化（Docker）、基础设施即代码（IaC） |
| **回滚机制** | 部署失败时快速恢复 | 蓝绿部署、金丝雀发布、版本标记 |
| **依赖管理** | 服务间依赖关系复杂 | 依赖图分析、按拓扑顺序部署 |
| **健康检查** | 如何判断部署成功 | 多维度健康检查（HTTP、TCP、自定义探针） |
| **零停机部署** | 不影响线上服务 | 滚动更新、流量切换 |

#### Q2: 介绍motion graphic agent是什么，架构是什么

**解答：**

Motion Graphic Agent是一种能够自动生成动态图形/动画的AI Agent。

**典型架构：**

```
用户输入（文本/语音）
    ↓
意图理解模块 → 提取动画需求（风格、时长、元素等）
    ↓
素材检索模块 → 从素材库获取图片/图标/字体
    ↓
脚本生成模块 → 生成动画时间轴和关键帧
    ↓
渲染引擎 → 合成最终视频/GIF
    ↓
输出结果
```

#### Q3: 项目用的是把skill直接塞进系统提示词中，如果skill太多，占用的上下文窗口太大，怎么处理？

**解答：**

**解决方案：**

1. **Skill检索机制**
   - 将Skill描述向量化存储
   - 根据用户查询动态检索相关Skill
   - 只加载Top-K相关Skill到Prompt

2. **Skill分层管理**
   - 基础Skill：常驻内存
   - 领域Skill：按需加载
   - 临时Skill：单次会话有效

3. **Skill摘要压缩**
   - 使用LLM对Skill描述进行摘要
   - 保留核心功能描述，去除冗余信息

4. **Skill组合优化**
   - 将相关Skill合并为复合Skill
   - 减少独立Skill数量

#### Q4: 如果两个skill实际内容不一样，但是description相似，导致agent每次加载错skill，怎么解决？

**解答：**

**解决方案：**

1. **优化Skill描述**
   - 增加区分性关键词
   - 明确标注适用场景和边界条件
   - 添加负面示例（什么情况下不该用）

2. **引入Skill示例**
   - 为每个Skill提供使用示例
   - 通过Few-shot方式帮助模型理解

3. **多路召回+重排序**
   - 召回多个候选Skill
   - 使用更精确的模型进行二次筛选

4. **用户反馈机制**
   - 记录Skill选择准确率
   - 根据反馈持续优化描述

| **记忆分层** | 区分短期记忆和长期记忆 |

#### Q16: Agent的"规划-执行-反思"闭环如何实现？

**解答：**

```
用户输入
    ↓
[规划阶段] → 生成执行计划
    ↓
[执行阶段] → 按计划调用工具/生成内容
    ↓
[反思阶段] → 检查结果是否符合预期
    ↓
    ├─ 符合 → 返回结果
    └─ 不符合 → 重新规划 → 循环
```

**关键点：**
- 设置最大迭代次数防止死循环
- 明确定义"完成"的判断标准
- 记录中间状态便于调试

#### Q17: 多轮对话中，记忆如何分级管理？缓存策略怎么设计？

**解答：**

**分级管理：**

| 级别 | 内容 | 存储 | 生命周期 |
|------|------|------|---------|
| L1 | 当前轮次 | 内存 | 单次请求 |
| L2 | 当前会话 | Redis | 会话期间 |
| L3 | 用户级记忆 | 向量DB | 长期 |
| L4 | 全局知识 | 知识库 | 永久 |

**缓存策略：**
- LRU：最近最少使用
- TTL：设置过期时间
- 重要性加权：重要信息优先保留

#### Q20: Agent的长记忆和短记忆之间，怎么做到协同工作的？两者的衔接逻辑是什么？

**解答：**

**协同流程：**

```
用户输入
    ↓
短期记忆检查 → 是否足够回答？
    ↓是              ↓否
直接回答    长期记忆检索
                ↓
         相关性筛选
                ↓
         融入短期记忆
                ↓
            生成回答
                ↓
         重要信息写入长期记忆
```

**衔接逻辑：**
- 触发机制：短期记忆不足时触发检索
- 优先级：短期记忆 > 高度相关的长期记忆
- 更新机制：重要信息从短期转入长期

#### Q25: Agent工具调用的完整业务流程是怎样的？

**解答：**

```
1. 意图识别 → 判断是否需要调用工具
2. 工具选择 → 从可用工具中选择合适的
3. 参数提取 → 从用户输入提取工具参数

7. 结果整合 → 将结果融入回答
```

#### Q28: Agent如何与MCP Server连接通信？

**解答：**

MCP（Model Context Protocol）通信流程：

```
Agent ←→ MCP Client ←→ MCP Server ←→ 外部资源
```

**连接方式：**
1. **Stdio**：本地进程通信
2. **SSE**：服务器推送事件
3. **HTTP**：RESTful API

**通信流程：**
1. Agent发送上下文请求
2. MCP Client转发到Server
3. Server访问资源（文件/数据库/API）
4. 返回结构化上下文
5. Agent整合上下文生成回答

#### Q54: 什么是RAG？它是怎么提升生成质量的？

**解答：**

#### Q59: 为什么要引入父子索引？BM25引入的原因是什么，比例是怎么设置的，整个具体流程是怎样的，有没有做rerank操作？

**解答：**

**父子索引：**

| 索引类型 | 作用 | 大小 |
|----------|------|------|
| 子索引 | 用于精确检索 | 小（如256 tokens） |
| 父索引 | 用于生成上下文 | 大（如2048 tokens） |

**BM25引入原因：**
- 向量检索擅长语义匹配
- BM25擅长精确关键词匹配
- 两者互补提高召回率

**混合比例：**
- 通常向量检索:关键词检索 = 70%:30%
- 根据场景调整

**流程：**
```
查询 → 向量化 + 分词 → 向量检索 + BM25检索 → 结果融合 → Rerank → Top-K
```

## 四、Transformer与注意力机制

#### Q89: Transformer中Attention的本质是什么？你能从数学角度简要解释一下吗？

**解答：**

**本质：** 加权求和，计算序列元素间的相关性。

**数学表达：**

```
Attention(Q, K, V) = softmax(QK^T / √d_k) × V

其中：
Q = X × W_Q  (Query)
K = X × W_K  (Key)
V = X × W_V  (Value)
```

**直观理解：**
- Q：我要查询什么
- K：我有什么信息
- V：信息的内容是什么

#### Q90: 为什么Attention要做缩放？

**解答：**

**原因：**

当d_k较大时，QK^T的点积结果会很大，导致softmax梯度消失。

**缩放因子√d_k的作用：**
- 控制点积的数值范围
- 保持方差稳定
- 避免梯度问题

#### Q91: Self-Attention和Cross-Attention在作用上有什么区别？

**解答：**

| 类型 | 输入 | 作用 |
|------|------|------|
| **Self-Attention** | Q=K=V（同一序列） | 学习序列内部关系 |
| **Cross-Attention** | Q来自一个序列，K/V来自另一个 | 学习两个序列间关系 |

**应用场景：**
- Self-Attention：编码器、解码器自注意力
- Cross-Attention：机器翻译（源语言→目标语言）

#### Q92: 如果序列长度非常长，Attention计算复杂度问题怎么解决？

**解答：**

**优化方案：**

| 方法 | 复杂度 | 原理 |
|------|--------|------|
| **稀疏Attention** | O(n) | 只计算局部注意力 |
| **线性Attention** | O(n) | 核函数近似softmax |
| **FlashAttention** | O(n²) | IO感知的分块计算 |
| **MQA/GQA** | - | 共享K/V，减少缓存 |

#### Q93: 在Agent多轮对话任务中，你觉得Attention的局限性体现在哪些方面？

**解答：**

**局限性：**

1. **上下文长度限制**
   - 长对话超出上下文窗口

2. **位置偏见**
   - 对长距离依赖建模能力有限

3. **计算成本高**
   - 长序列计算量大

4. **信息遗忘**
   - 早期信息可能被稀释

#### Q94: 在长上下文对话里，模型为什么容易出现信息遗忘？

**解答：**

**原因：**

1. **Softmax稀释**
   - 注意力权重被大量token分摊

2. **位置编码限制**
   - 长距离位置信息衰减

3. **训练数据偏差**
   - 训练时短序列为主

#### Q95: 目前常见有哪些机制可以缓解这种问题？

**解答：**

| 机制 | 说明 |

- 追求最佳效果且有资源：RLHF

| **计算成本** | 高 | 低 |

#### Q115: 项目提到了多个工具调用链路，调度策略是如何设计的？

| 策略 | 说明 |
|------|------|
| **串行执行** | 有依赖关系的工具按序执行 |
| **并行执行** | 无依赖的工具并行执行 |

#### Q122: 分布式令牌桶限流、漏桶限流，还有滑动窗口算法限流的原理？

**解答：**

**令牌桶：**

```
以固定速率产生令牌放入桶中
请求需要获取令牌才能执行
桶满时丢弃令牌
```

**漏桶：**

```
请求先进入桶中排队
以固定速率处理请求
桶满时丢弃新请求
```

**滑动窗口：**

```
将时间划分为小窗口
统计当前窗口内的请求数
超过阈值则限流
```

**对比：**

| 算法 | 突发流量 | 平滑性 | 实现复杂度 |
|------|----------|--------|------------|
| 令牌桶 | 允许 | 较好 | 中 |
| 漏桶 | 不允许 | 最好 | 中 |
| 滑动窗口 | 不允许 | 一般 | 低 |

#### Q123: 滑动窗口的数据结构会包含哪些字段？滑动窗口对比令牌桶有什么缺点？

#### Q161: 讲讲python的gil（全局解释器锁）是什么？

#### Q163: Go语言的GMP调度模型是怎样的？

**解答：**

**GMP模型：**

| 组件 | 说明 |
|------|------|
| **G（Goroutine）** | 协程，用户态轻量级线程 |
| **M（Machine）** | 系统线程，执行Goroutine |
| **P（Processor）** | 逻辑处理器，维护Goroutine队列 |

**调度流程：**

```
Goroutine创建 → 放入P的本地队列 → M获取G执行
                    ↓
            本地队列满 → 放入全局队列
```

#### Q164: GMP模型中的Machine（M）代表什么？

**解答：**

**M（Machine）：**

- 操作系统线程（OS Thread）
- 负责实际执行Goroutine的代码
- M与P绑定后才能执行G
- 数量通常与CPU核心数相关

#### Q165: Processor（P）上的本地队列长度通常是多少？

**解答：**

**默认长度：**

- 本地队列：256个Goroutine
- 全局队列：无限制

**设计原因：**
- 避免频繁访问全局队列（需要锁）
- 提高调度效率

## 十四、计算机网络

#### Q167: 讲讲tcp的三次握手

**解答：**

**三次握手过程：**

```
客户端                    服务器
  |                         |
  |---- SYN,seq=x --------->|  （第一次：客户端请求建立连接）
  |                         |
  |<--- SYN,seq=y,ACK=x+1 --|  （第二次：服务器确认并请求）
  |                         |
  |---- ACK=y+1 ----------->|  （第三次：客户端确认）
  |                         |
  |    连接建立，开始传输    |
```

**为什么是三次：**
- 防止历史重复连接
- 同步双方初始序列号
- 确认双方收发能力
